# 四项接口自动化 — 流程逻辑说明

---

## 总览

| # | 自动化 | 工具 | 核心难点 | 接口数 |
|:--|--------|------|---------|:-----:|
| 一 | 创建知识库审批 | Apifox | 创建接口不返回 ID | 4 |
| 二 | Wiki 飞书云文档上传 | Apifox 数据驱动 | 单文件参数串联 | 6 |
| 三 | 飞书云文档上传 | Apifox | BFS 遍历文件夹树 | 7 |
| 四 | OSS 批量上传本地文件 | Python | PUT 协议限制 + 并发 | 5 |

## 一、创建知识库审批（Apifox）

### 业务逻辑

点击「创建知识库」→ 填写信息 → 提交 → 切换到审核管理 → 找到审批记录 → 审批通过

### 痛点

创建接口 `POST /api/kb/project/add` 响应 `data` 为 `null`，不返回任何 ID。必须通过审批列表间接拿到 `task_id`，才能调审批接口。

### 接口筛选

F12 抓到了 10 个接口，但真正需要自动化的只有 4 个：

| 接口 | 作用 | 保留？ |
|------|------|:----:|
| available-list | 加载标签体系下拉框 | ❌ UI 渲染 |
| match-by-dept | 按部门匹配标签 | ❌ UI 渲染 |
| user/list ×3 | 搜索管理员 | ❌ UI 渲染 |
| **project/add** | 提交创建申请 | ✅ |
| **notifyList** | 查审批列表，拿 task_id | ✅ |
| **approve/audit** | 审批通过 | ✅ |
| **audit/detail** | 验证审批结果 | ✅ |

### 流程

```
① project/add    → 提交创建申请
   请求体 name/remark 用环境变量 {{kb_name}}
   响应 {"data": null}  ← 不返回任何 ID

② notifyList     → 查"我审核的" KB_PROJECT 列表
   列表按时间倒序，刚创建的在第一条
   → 提取 task_id = records[0].task_id

③ approve/audit  → 审批通过
   task_id + approve_result=1 + comment

④ audit/detail   → 验证结果
   确认 audit_status=2（已通过）、audit_result=1
```

### 关键发现：参数串联靠"取第一条"

创建后立刻查列表，按时间倒序，第一条就是要审批的。不需要复杂的名称匹配或时间匹配。越简单越可靠。

## 二、Wiki 飞书云文档上传（Apifox · 数据驱动）

### 业务逻辑

预置飞书文档信息（token/name/file_type）在环境变量中 → 查询文档详情 → 查重 → 上传 → 推送解析 → 验证转换 → 生成下载链接

### 和"飞书文件夹上传"的区别

| | Wiki 文档上传 | 飞书文件夹上传 |
|---|---|---|
| **场景** | 导入单个/多个已知文档 | 导入整个文件夹（含嵌套） |
| **遍历逻辑** | 无，数据预置在环境变量 | BFS 逐层扫描 |
| **文件来源** | 环境变量 `feishu_file_list` 预填 | 从飞书 API 实时发现 |
| **上传** | 单文件 file/upload | 根层 file/upload + 文件夹 dir/upload |
| **轮询** | 查一次 → 条件判断 | Loop 15 次逐步检查 |
| **数据驱动** | ✅ CSV/JSON 批量跑不同文档 | 无 |

### 适用场景

知道具体要传哪些飞书文档（已抓取到文档的 token/name/file_type），不需要扫描整个文件夹树。配合 Apifox 数据驱动功能，可以用 CSV 文件批量上传不同文档到不同知识库。

### 流程

```
① children_dir    → 校验飞书链接有效性
② dir_files       → 获取文档的 token、name、repeat_name
                     提取 data[0].name → filename
                     提取 data[0].token → token
                     提取 data[0].repeat_name → repeat_name
③ 查重             → 用提取到的 token + filename
④ 上传文件         → token + filename + is_replace(=repeat_name)
                     提取 data.fileIds → fileIds
⑤ 推送文件状态     → 用 fileIds 触发后台解析
⑥ 延时             → 等待解析
⑦ 查询文件列表     → 提取 rows[0].cover → cover
                     断言 rows[0].convert_status == COMPLETED
⑧ 条件分支         → cover 不为空？
    └─ 生成预签名下载链接
```

### 关键设计

**① 只取 `data[0]`** — 每次查的是单个文件，永远取第一个元素，不需要 ForEach。

**② `is_replace` 自动覆盖** — `repeat_name` 字段表示是否重名，上传时用这个值让系统自动决定覆盖。

**③ 数据驱动** — `feishu_file_list` 预置在环境变量中：
```json
[
  {"token":"xxx", "filename":"肺癌...", "is_replace":true, "file_type":"DOCX"},
  {"token":"xxx", "filename":"唐林.jpg", "is_replace":true, "file_type":"FILE"}
]
```

## 三、飞书云文档上传（Apifox · BFS 遍历）

整个流程分两大阶段：**扫描阶段**（搞清楚飞书文件夹里有什么）和**上传阶段**（把搞清楚的东西传到知识库）。

---

### 阶段一：扫描 — 搞清楚文件夹里有什么

飞书链接对应的文件夹树示例：

```
根链接
├── 测试1/                      ← 文件夹
│   ├── 测试1-1/                ← 子文件夹
│   │   └── 测试1-1-1/          ← 孙文件夹
│   │       ├── 指导原则.pdf
│   │       └── 报销制度.pdf
│   ├── GenAI报告.pdf           ← 文件（挂在测试1下）
│   └── 肺鳞癌.pdf
├── 测试2/                      ← 文件夹
│   └── 测试2-1/
│       ├── 测试2-1-1/
│       └── 先秦时期.docx
├── 肺癌.docx                   ← 根层文件
└── 唐林.jpg                    ← 根层文件
```

#### 第 1 步：查根层有什么文件夹

把飞书链接发给 `children_dir`，服务器返回根层有两个文件夹——测试 1 和测试 2。提取器把这两个文件夹的 token 存到临时变量 `tokens`。

#### 第 2 步：查根层有什么文件

同一个链接发给 `dir_files`（多传 `project_id`，因为需要判断文件是否已经导入过）。服务器返回两个文件——肺癌.docx 和唐林.jpg。自定义脚本把这俩文件存到**环境变量** `feishu_file_list`（用环境变量而非临时变量，因为后面第 5 步上传文件时还要用，临时变量到那一步已经过期了）。

#### 第 3 步：BFS — 一层一层往下挖

这是整个自动化最核心的一步。`tokens = [测试1, 测试2]`，外面包一个 Loop（写 10 次，实际 3 层就结束），里面是**条件分支 + ForEach + 三个脚本 S1/S2/S3**。

**Loop 第 1 轮**（`tokens = [测试1, 测试2]`）：

```
├─ S1（清空累积桶）：清空 accumulated_next_tokens，避免上轮残留数据干扰
│
├─ 条件分支：tokens 不为空 → 进入
│
├─ ForEach tokens：
│   │
│   ├─ [测试1]
│   │   接口 A（children_dir）：查测试1下的子文件夹 → [测试1-1]
│   │      提取器 → next_tokens
│   │      自定义脚本 → 把 next_tokens 追加进 accumulated_next_tokens（不是覆盖，是累加）
│   │
│   │   接口 B（dir_files）：查测试1下的文件 → [GenAI.pdf, 肺鳞癌.pdf]
│   │      提取器 → subFiles
│   │      自定义脚本 → 在 folderList 树里找到测试1节点，挂载 children
│   │
│   ├─ [测试2]
│   │   接口 A（children_dir）：查测试2下的子文件夹 → [测试2-1]
│   │      追加进 accumulated_next_tokens
│   │
│   │   接口 B（dir_files）：查测试2下的文件 → []（空的）
│   │
│   ForEach 结束。accumulated_next_tokens = [测试1-1, 测试2-1]
│
├─ S2（合并）：next_tokens = accumulated_next_tokens → [测试1-1, 测试2-1]
│
└─ S3（轮换）：tokens = next_tokens，清空 next_tokens
```

**Loop 第 2 轮**（`tokens = [测试1-1, 测试2-1]`）：

```
S1 → 清空 accumulated_next_tokens
ForEach：
  测试1-1 → 没子文件夹、没文件
  测试2-1 → 子文件夹 [测试2-1-1]（追加进桶）、文件 [先秦时期.docx]（挂载到树）
accumulated_next_tokens = [测试2-1-1]
S2 → next_tokens = [测试2-1-1]
S3 → tokens = [测试2-1-1]
```

**Loop 第 3 轮**（`tokens = [测试2-1-1]`）：

```
ForEach：没子文件夹、没文件
accumulated_next_tokens = []
S2 → next_tokens = []
S3 → tokens = []
```

**Loop 第 4 轮**：条件分支判断 `tokens` 为空 → 跳过 ForEach → Loop 结束。

**三个脚本的职责：**

| 脚本 | 位置 | 作用 |
|------|------|------|
| S1 | ForEach 前面 | 每轮 BFS 开始前清空累积桶，避免残留数据干扰 |
| 接口 A 自定义脚本 | ForEach 内部 | 将每次迭代查到的子文件夹 token **累加**到累积桶（解决 ForEach 覆盖问题） |
| 接口 B 自定义脚本 | ForEach 内部 | 将子文件夹和子文件挂载到 folderList 树中，层层构建完整目录树 |
| S2 | ForEach 后面 | 将累积桶中本层收集到的全部子文件夹取出，设为下一层的 next_tokens |
| S3 | ForEach 后面 | 轮换：tokens = next_tokens，清空 next_tokens，准备进入下一层 |

---

### 阶段二：上传 — 把搞清楚的东西传到知识库

#### 第 4 步：查重

把根层的文件（来自 `feishu_file_list`）+ 根层的文件夹（来自 `folderList`）拼在一起发给 `repetition` 接口。只查根层，子文件夹里的文件不在此处处理，它们会在第 6 步随文件夹树一起提交。

#### 第 5 步：上传根层文件

只传根层的文件（肺癌.docx + 唐林.jpg）。子文件夹内的文件不单独上传。服务器返回 `fileIds: [7364, 7365]`，提取器存到 `file_ids`。

#### 第 6 步：上传完整文件夹树

将 `folderList` 树递归转成接口要求的嵌套格式——文件夹 `type="DIRECTORY"`，文件 `type` 取自身的 `file_type`。所有子文件夹内的文件都嵌套在树的 `children` 中一次性提交。

#### 第 7 步：轮询等解析完成

文件上传后后台异步解析。Loop 15 次，每次等 3 秒：

- **接口 D（查文件列表）**：获取每个文件的 `convert_status`。自定义脚本筛出 PENDING/IN_PROGRESS 的 → `pending_file_ids`；计算布尔值 `is_all_complete`（全部 COMPLETED/FAILED 时为 true）。同时提取 `coverArray`，存为集合变量供下一步使用
- **条件分支**：`is_all_complete == false`？→ 调接口 E 推送状态，继续等待。`true`？→ 退出循环

#### 第 8 步：生成下载链接

ForEach 遍历 `coverArray`，不为 null 的调接口生成预签名下载 URL。

#### 第 9 步：汇总报告

脚本将 `file_ids`、`coverArray`、最后一次 list 结果拼成报告，逐文件打印解析结果和封面状态。

---

### 一句话总结

```
前两步看根层 → BFS 层层递进构建完整文件夹树
→ 查重 → 根层文件单独传 → 整棵文件夹树一起传
→ 轮询等到全部解析完 → 每个文件生成下载链接 → 打印报告
```

### 数据流转的关键变量

| 变量名 | 来源 | 内容 | 作用域 |
|--------|------|------|--------|
| `tokens` | children_dir 提取 | 当前层所有文件夹 token 数组 | 临时 |
| `folderList` | children_dir 提取 + 递归挂载 | 完整文件夹树（含嵌套 children） | 临时 |
| `feishu_file_list` | dir_files 自定义脚本 | 根层文件列表 | 环境 |
| `file_ids` | file/upload 提取 | 上传成功的文件 ID 数组 | 临时 |
| `is_all_complete` | list 轮询脚本计算 | 所有文件是否解析完成 | 临时 |
| `coverArray` | list 轮询脚本提取 | 每个文件的封面 key | 集合 |
| `accumulated_next_tokens` | BFS 累积脚本 | 下一层 tokens 累积池 | 临时 |

### 搭建过程中修复的典型问题

| 问题 | 现象 | 根因 | 解决 |
|------|------|------|------|
| BFS 丢子文件夹 | 只遍历到第一层 | ForEach 覆盖 `next_tokens` | 累加到累积变量 |
| 文件夹上传 400 | `知识库文件文件不是文件夹` | type 字段误判 | 根据 `file_type` 正确区分 DIRECTORY/FILE |
| 第二步 500 | `Session ID mismatch` | 缺阿里云 WAF Cookie `acw_tc` | 启用全局 Cookie 管理 |
| 轮询过早退出 | 文件未解析完就跳出 | Loop 写死 3 次不够 | 改为 15 次 × 3s |
| 后置脚本崩溃 | `Cannot read properties of null` | `data` 为 null 时 `.map()` | 前置 `if (!response.data) return;` |

### 运行效果

| 对比维度 | 手动操作 | 自动化 |
|----------|:-------:|:-----:|
| 耗时（3 层 10 文件） | ~5 分钟 | ~90 秒 |
| 错误发现 | 肉眼逐条看 | 断言校验 + 汇总报告 |
| 可重复性 | 每次手动粘贴 | 一键运行 |

## 四、OSS 批量上传本地文件（Python）

### 业务逻辑

选择本地多个文件 → 上传到知识库 → 后台解析 → 生成封面

### 痛点

Apifox 的 PUT 请求只能串行上传。PUT 是 HTTP 协议语义——一个请求体就是一个文件，无法像 POST 的 `multipart/form-data` 打包多个文件。

### 为什么用 Python

```
Apifox:  ForEach 拿到预签名 URL → 一个一个 PUT → 中途断一次全流程报废
Python:  3 线程同时 PUT → 失败自动重试 → 全部跑完才退出
```

### 流程

```
① scan_files()      扫描 D:\yzj_test\ → N 个文件路径
② create_upload_tasks()  批量创建上传任务 → 每个文件生成 fileId
③ [并发池 3 线程]
   ├─ 获取预签名 URL      POST /api/oss/generate_pre_upload_signature
   ├─ PUT 上传到 OSS       阿里云 OSS 预签名链接
   ├─ 注册文件             POST /api/kb/file/upload
   └─ 更新任务状态        POST /api/uploadTask/status → SUCCESS
④ push_file_status()  推送状态 → 触发解析
⑤ wait_for_conversion()  轮询 → 等待 convert_status=COMPLETED
⑥ generate_download_url()  生成下载链接
```

### 关键发现

**① 并发数 3 是最优值** — 公司内网服务器：5 并发时连接重置率明显上升，3 并发是实测的最优平衡点。

**② 为什么 PUT 只能是 binary** — OSS 预签名 URL 里签名和文件名绑定，一个 URL 只能传一个文件。

| | POST | PUT |
|---|---|---|
| HTTP 语义 | 创建资源 | **替换**指定 URL 的完整实体 |
| 请求体格式 | `multipart/form-data`，可含多个文件 | **单一二进制流（binary）** |
| 幂等性 | 不幂等 | 幂等，重复 PUT 同一内容结果不变 |
| 与 OSS 的匹配 | 不合适，签名无法覆盖多个文件 | 语义完美：URL = 资源位置，PUT = 写入 |

**③ 错峰提交** — 线程间隔 0.5s 启动，避免同时打服务器。

**④ 重试机制** — `api_post` 捕获 `ConnectionError`/`Timeout`/`HTTPError`，自动重试 3 次，间隔递增。内网环境下间歇性连接重置（ConnectionResetError 10054）是常态，没有重试 8 个文件能失败 3-4 个。

### 效果对比

| 维度 | Apifox 手动 | Python 脚本 |
|------|:---------:|:--------:|
| 8 文件总耗时 | ~5 分钟 | ~40 秒 |
| 失败重试 | 中断后手动排查 | 自动重试 |
| 操作步骤 | 多个参数 + 等待 | 粘贴认证信息 + 回车 |

## 四者的共性规律

```
┌─────────────────────────────────────────────────────┐
│  1. F12 抓完整流程 → 区分 UI 接口和业务接口          │
│  2. 找参数串联关系（哪两个接口通过什么字段衔接）     │
│  3. 固定值写死、变化值用环境变量                     │
│  4. 循环/轮询的次数不是拍脑袋，是实测出来的          │
│  5. 遇到坑先看真实请求（curl），不是看 API 文档      │
│  6. 越简单的方案越可靠（取第一条 > 复杂匹配）        │
└─────────────────────────────────────────────────────┘
```

## 四个流程的核心收获

1. **先抓包再写脚本** — 浏览器 F12 的 curl + 响应是最好的接口文档，比 API 文档更可靠
2. **理解状态管理** — Apifox 的变量有临时/环境/集合三种作用域和生命周期，选错类型会导致数据串扰
3. **ForEach 覆盖陷阱** — 任何在 ForEach 内 set 的临时变量都会被下一次迭代覆盖，必须额外累积
4. **WAF Cookie 不可忽视** — 阿里云 WAF 的 `acw_tc` Cookie 缺失会导致网关拦截，表现为不明确的 500
5. **数据驱动是效率倍增器** — 用 CSV 批量替换环境变量，一次运行覆盖多个测试场景
6. **工具选型看瓶颈** — Apifox 做不了的（PUT 并发）就该换 Python；能做的就别换

## 四个自动化的互补关系

```
创建知识库审批 ───→ 创建知识库（业务入口）
     │
     └──→ Wiki 文档上传 ───→ 导入已知文档（数据驱动）
              │
              └──→ 文件夹上传 ───→ 导入整个文件夹（BFS 遍历）
                       │
                       └──→ OSS 上传 ───→ 本地文件上云（并发）

一条知识库管理全链路：创建 → 飞书批量导入 → 本地批量导入
```

## 环境配置速查

**Apifox 环境变量**（三个自动化共用）：

| 变量名 | 说明 |
|--------|------|
| `authorization` | Bearer Token |
| `KB_SESSION_ID` | Session Cookie |
| `project_id` | 知识库项目 ID |
| `feishu_url` | 飞书文件夹链接（文件夹上传用） |
| `Wikifeishu_url` | 飞书文档链接（Wiki 上传用） |
| `kb_name` | 知识库名称（审批用） |

**全局配置**：
- 启用「全局 Cookie 参数」— 自动管理 WAF Cookie
- 每个 HTTP 接口 Header：`Authorization` + `Cookie` + `Content-Type`

**Python 依赖**（OSS 脚本）：
```bash
pip install requests
```
