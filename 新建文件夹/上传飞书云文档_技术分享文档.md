# 飞书云文档上传接口自动化

## 知识库管理后台 · 测试自动化实践

---

## 一、背景与痛点

公司知识库管理后台支持通过飞书链接导入云文档。导入一个飞书文件夹链接后，后台需要完成：

1. 递归扫描整个飞书文件夹树（不限层级深度）
2. 逐个发现文件夹和文件，记录 token
3. 查重 → 上传文件 → 上传文件夹结构 → 轮询等待解析 → 生成下载链接

此前每次发版回归，需要手动在页面上粘贴飞书链接、等待导入完成、再逐个校验文件解析结果。一个含 3 层嵌套、10 个文件的链接，手动操作约 5 分钟，且中间任何一步出错都无感知。

---

## 二、接口链路总览

整个导入流程涉及 **4 个核心接口**，按调用顺序：

| 步骤 | 接口 | 作用 | 调用次数 |
|------|------|------|---------|
| 遍历 | `POST /api/kb/feishu/children_dir` | 查文件夹下的子文件夹 | 1 + N（每个文件夹 1 次） |
| 遍历 | `POST /api/kb/feishu/dir_files` | 查文件夹下的文件 | 1 + N（每个文件夹 1 次） |
| 上传 | `POST /api/kb/file/repetition` | 查重文件名 | 1 |
| 上传 | `POST /api/kb/file/upload` | 上传根层文件 | 1 |
| 上传 | `POST /api/kb/dir/upload` | 上传完整文件夹树 | 1 |
| 轮询 | `POST /api/kb/file/list` | 查询文件解析状态 | 多次（轮询） |
| 轮询 | `POST /api/kb/status` | 推送文件状态触发解析 | 多次（轮询） |
| 结果 | `POST /api/oss/generate_pre_download_signature` | 生成下载链接 | 每个文件 1 次 |

## 三、核心难点：BFS 层序遍历文件夹树

### 问题

飞书文件夹可能包含任意深度的嵌套结构。示例：

```
根链接
├── 测试1/                    ← 文件夹
│   ├── 测试1-1/              ← 子文件夹
│   │   ├── 测试1-1-1/        ← 孙文件夹
│   │   │   ├── 指导原则.pdf  ← 文件
│   │   │   └── 报销制度.pdf
│   │   ├── 专家共识.docx
│   │   └── 竞品对比.xlsx
│   ├── GenAI报告.pdf         ← 文件（挂在测试1下）
│   └── 肺鳞癌.pdf
├── 测试2/                    ← 同层另一个文件夹
│   └── 测试2-1/
│       ├── 测试2-1-1/
│       └── 先秦时期.docx
├── 肺癌.docx                 ← 根层文件
└── 唐林.jpg                  ← 根层文件
```

### 方案：BFS 层序遍历

每一层拿到 tokens → ForEach 逐个查子文件夹和子文件 → 收集下一层 tokens → 继续。用累积变量解决 Apifox ForEach 覆盖问题。

```
第1轮 tokens = [测试1, 测试2]
  ForEach 测试1 → 子夹 [测试1-1], 文件 [GenAI.pdf, 肺鳞癌.pdf]
  ForEach 测试2 → 子夹 [测试2-1], 文件 []
  合并 → 下一轮 tokens = [测试1-1, 测试2-1]

第2轮 tokens = [测试1-1, 测试2-1]
  ForEach 测试1-1 → 子夹 [测试1-1-1], 文件 [专家共识.docx, 竞品.xlsx]
  ForEach 测试2-1 → 子夹 [测试2-1-1], 文件 [先秦时期.docx]
  合并 → 下一轮 tokens = [测试1-1-1, 测试2-1-1]

第3轮 tokens = [测试1-1-1, 测试2-1-1]
  ForEach 测试1-1-1 → 子夹 [], 文件 [指导原则.pdf, 报销制度.pdf]
  ForEach 测试2-1-1 → 子夹 [], 文件 []
  合并 → 下一轮 tokens = []

tokens 为空 → BFS 结束
```

### 关键代码：累积 token 防覆盖

Apifox 的 ForEach 每次迭代都会覆盖临时变量，需要在后置脚本中主动累积：

```javascript
// 读取当前查到的子文件夹 tokens
let nextTokens = JSON.parse(pm.variables.get("next_tokens") || "[]");

// 读取累积变量
let accumulated = JSON.parse(pm.variables.get("accumulated_next_tokens") || "[]");

// 累加而非覆盖
accumulated = accumulated.concat(nextTokens);
pm.variables.set("accumulated_next_tokens", JSON.stringify(accumulated));
```

## 四、数据流转的关键变量

| 变量名 | 来源 | 内容 | 作用域 |
|--------|------|------|--------|
| `tokens` | children_dir 提取 | 当前层所有文件夹 token 数组 | 临时 |
| `folderList` | children_dir 提取 + 递归挂载 | 完整文件夹树（含嵌套 children） | 临时 |
| `feishu_file_list` | dir_files 自定义脚本 | 根层文件列表 | 环境 |
| `file_ids` | file/upload 提取 | 上传成功的文件 ID 数组 | 临时 |
| `is_all_complete` | list 轮询脚本计算 | 所有文件是否解析完成 | 临时 |
| `coverArray` | list 轮询脚本提取 | 每个文件的封面 key | 集合 |
| `accumulated_next_tokens` | BFS 累积脚本 | 下一层 tokens 累积池 | 临时 |

## 五、关键修复记录

实际搭建过程中遇到的典型问题：

| 问题 | 现象 | 根因 | 解决 |
|------|------|------|------|
| BFS 丢子文件夹 | 只遍历到第一层，深层文件夹遗漏 | ForEach 覆盖 `next_tokens` | 累加到 `accumulated_next_tokens` |
| 文件夹上传 400 | `知识库文件文件不是文件夹` | `type` 字段判断错误，文件夹被标成 FILE | 根据 `file_type` 字段正确区分 DIRECTORY/FILE |
| 第二步 500 | `Session ID mismatch` | 缺阿里云 WAF Cookie `acw_tc` | 启用 Apifox 全局 Cookie 管理，不手写死 Cookie Header |
| 轮询过早退出 | 文件未解析完就跳去生成下载链接 | Loop 写死 3 次不够 | 改为 15 次 × 3s 延时 |
| 汇总脚本报错 | `Cannot read properties of null` | `data` 为 null 时 `.map()` 崩溃 | 前置 `if (!response.data) return;` 保护 |

## 六、环境配置

在 Apifox 自动化测试场景中，配置 4 个环境变量：

| 变量名 | 类型 | 说明 |
|--------|------|------|
| `authorization` | string | 登录后抓取的 Bearer Token |
| `KB_SESSION_ID` | string | 登录后 Cookie 中的 Session ID |
| `project_id` | string | 目标知识库项目 ID |
| `feishu_url` | string | 待导入的飞书文件夹链接 |

### 全局配置

- 启用「全局 Cookie 参数」——自动管理 `acw_tc` 等 WAF Cookie
- 每个 HTTP 接口的 Header 统一配置：`Authorization` + `Cookie` + `Content-Type`

## 九、核心收获

1. **先抓包再写脚本** — 浏览器 F12 的 curl + 响应是最好的接口文档，比 API 文档更可靠
2. **理解状态管理** — Apifox 的变量有临时/环境/集合三种作用域和生命周期，选错类型会导致数据串扰
3. **ForEach 覆盖陷阱** — 任何在 ForEach 内 set 的临时变量都会被下一次迭代覆盖，需要额外做累积
4. **WAF Cookie 不可忽视** — 阿里云 WAF 的 `acw_tc` Cookie 缺失会导致网关拦截，表现为不明确的 500 错误
