## 一、上传本地文件（Python）

### 卡点 1：Apifox 的 PUT 为什么不能批量上传

**现象**：Apifox 里 ForEach 遍历文件 + PUT 请求，30 个文件只能一个一个传，中间断一次全废。

**排查过程**：不是 Apifox 的 bug。用 curl 重现 PUT 请求，发现请求体就是文件本身的二进制流，没有 key=value 结构。翻 HTTP 协议规范才确认——PUT 的语义是"替换指定 URL 的完整实体"，一个 PUT 请求只能携带一个文件。

**深层原因**：阿里云 OSS 的预签名 URL 里，签名 `Signature` 是对文件名 + 过期时间 + AccessKey 的加密结果。URL 和文件名绑死了，不可能拿一个 URL 传另一个文件。

**解决方案**：换 Python。用 `ThreadPoolExecutor` 开启 3 个线程，每个线程独立拿到自己文件的预签名 URL，并发 PUT。失败自动重试，总耗时从 5 分钟降到 40 秒。

### 卡点 2：ConnectionResetError 10054

**现象**：跑 8 个文件，每次都有 3-4 个报 `ConnectionResetError(10054, '远程主机强迫关闭了一个现有的连接')`。

**排查过程**：降低并发数从 5 到 3，错误率明显下降。加 3 次自动重试、间隔递增（3s、6s），问题基本消失。

**根因**：公司内网服务器承载能力有限，无法同时处理 5 个并发连接。`api_post` 的 JSON 请求和 OSS PUT 的二进制流请求报的错一样，说明是服务器端主动断连。

**解决方案**：
```python
# 1. 并发数降到 3
MAX_WORKERS = 3

# 2. api_post 自动重试
for retry in range(3):
    try:
        resp = requests.post(...)
        resp.raise_for_status()
        return resp.json()
    except (ConnectionError, Timeout, HTTPError) as e:
        if retry < 2:
            time.sleep((retry + 1) * 3)  # 3s, 6s 递增
            continue
        raise

# 3. OSS PUT 独立重试（含中文/特殊字符文件名更易触发）
for retry in range(3):
    try:
        with open(file_path, "rb") as fh:
            requests.put(upload_url, data=fh, timeout=300)
        break
    except Exception:
        if retry == 2: raise
        time.sleep(3)

# 4. 线程错峰提交
for i, (fp, task) in enumerate(zip(file_paths, tasks), 1):
    pool.submit(upload_single_file, fp, task, i)
    time.sleep(0.5)  # 避免同时打服务器
```

### 卡点 3：特殊字符文件名导致上传失败

**现象**：`融合LSTM与XGBoost的时间序列预测 ！.docx`（含全角感叹号 `！`）每次第一个连接就被 OSS 重置。

**根因**：预签名 URL 中的文件名经过 URL 编码，全角字符 `！`（U+FF01）在编码后与半角 `!` 不同，OSS 校验签名时不匹配。

**解决方案**：在 `api_post` 和 OSS PUT 里都加 3 次重试 + 递增间隔。重试能过，不需要改文件名。

## 二、上传飞书云文档（Apifox）

### 卡点 1：BFS 遍历丢子文件夹

**现象**：飞书文件夹有 3 层嵌套，但自动化只遍历了第一层，深层文件夹全部遗漏。

**排查过程**：在 ForEach 里打印 `next_tokens`，发现每次循环只保留最后一个文件夹的结果。比如根目录有测试 1 和测试 2：
- 循环 1 查测试 1 → `next_tokens = [测试1-1]`
- 循环 2 查测试 2 → `next_tokens = [测试2-1]` ← 把前面的覆盖了

Apifox 的 ForEach 每次迭代前都会重新设置临时变量，如果不做累积，就只剩最后一次的结果。

**解决方案**：引入 `accumulated_next_tokens` 累积变量。

```javascript
// 接口 A（ForEach 内的 children_dir）后置脚本中：
// 不是 pm.variables.set("next_tokens", value)
// 而是累加：
let accumulated = JSON.parse(pm.variables.get("accumulated_next_tokens") || "[]");
accumulated = accumulated.concat(nextTokens);
pm.variables.set("accumulated_next_tokens", JSON.stringify(accumulated));

// ForEach 结束后，合并脚本：
var accumulated = pm.variables.get("accumulated_next_tokens") || "[]";
pm.variables.set("next_tokens", accumulated);
```

**启示**：任何在 ForEach 内 set 的临时变量都会被下次迭代覆盖。要么用环境变量（会跨请求保留），要么手动累加。

### 卡点 2：第二步接口 500

**现象**：`POST /api/kb/feishu/dir_files` 返回 `{"code":"B0000","msg":"Session ID mismatch"}`。但 Cookie 和 JWT 里的 `sid` 完全一致。

**排查过程**：
1. 先怀疑 Cookie 重复（有硬编码的 `KB_SESSION_ID=Ak2Oau...`），删掉——无效
2. 再怀疑 `project_id` 类型不对（传了字符串 `"275"` 而非数字），改成数字——无效
3. 对比浏览器 F12 抓到的 curl 和 Apifox 发的 curl，发现浏览器多了一个 Cookie：`acw_tc=6585...`

**根因**：阿里云 WAF 网关种的 `acw_tc` Cookie。浏览器自动处理 `Set-Cookie` 所以有，Apifox 手动写了 `Cookie` Header 后就不会自动更新。

**解决方案**：不在 Header 里手写死 `Cookie: KB_SESSION_ID=xxx`，而是启用 Apifox 的「全局 Cookie 参数」。这样每次 `Set-Cookie` 都会被自动捕获并在后续请求中带上。

**启示**：接口报 `Session ID mismatch` 时，先对比 curl 的 Cookie 有哪些，再对比 Apifox 发了哪些。缺一个都不行。

### 卡点 3：文件夹上传 400

**现象**：`POST /api/kb/dir/upload` 返回 `{"code":"B0702","msg":"知识库文件文件不是文件夹"}`。

**根因**：`folderList` 树构建时，BFS 挂载的 children 混入了文件节点（从 `dir_files` 拿到的）。`build` 函数对没有 `children` 字段的节点默认判为 `"FILE"`，但文件夹也可能暂时没有 children（还没遍历到子层）。

**解决方案**：判断类型用 `file_type` 字段而非 `children` 字段。

```javascript
// 错误写法：
type: i.children !== undefined ? "DIRECTORY" : "FILE"

// 正确写法：
type: (i.file_type && i.file_type !== "FOLDER") ? i.file_type : "DIRECTORY"
```

**启示**：文件夹和文件的判断标准是接口返回的 `file_type` 字段，不能靠"有没有 children"推断。

### 卡点 4：轮询 Loop 写死 3 次不够

**现象**：文件还没解析完就进入生成下载链接的步骤，拿到的 cover 全是 null。

**排查过程**：看日志发现 `is_all_complete` 在第 3 次循环后仍然是 false，但 Loop 已经结束了。文件从上传到解析完成大概需要 30-45 秒。

**根因**：早期脚本 Loop 写死 3 次 × 1s 延时 = 最多等 3 秒，根本不够。

**解决方案**：Loop 改为 15 次 × 3s 延时 = 最多等 45 秒。配合 `is_all_complete` 布尔判断，提前完成就自然退出。

### 卡点 5：后置脚本在接口报错时崩溃

**现象**：接口返回 500 时，后置脚本报 `Cannot read properties of null (reading 'map')`，所有后续步骤都不执行。

**根因**：`response.data.map(...)` 在 `data` 为 null 时崩溃，整个测试场景中断。

**解决方案**：所有使用 `response.data` 的后置脚本，开头加保护：
```javascript
if (response.code !== "00000" || !response.data) {
  pm.test("接口响应异常", () => {
    pm.expect(response.code).to.equal("00000");
    pm.expect(response.data).not.to.be.null;
  });
  return;
}
```

### 卡点 6：数据驱动导致的困扰

**现象**：在 Apifox 中配置 Wiki 上传的数据驱动时，分不清 `Wikifeishu_url` 应该放在环境变量还是测试数据中。

**排查过程**：Apifox 的数据驱动机制要求测试数据中的变量名必须与环境变量名匹配。如果变量名不一致，运行时会取不到值。

**解决方案**：
1. 环境变量中定义变量名（留空值）
2. 测试数据 CSV 中按相同列名填入实际值
3. 数据驱动执行时，Apifox 自动用 CSV 的值替换环境变量

## 三条方法论

**1. 需求分析**

F12 抓完整流程 → 筛掉 UI 渲染接口（下拉框、搜索框、表格分页）→ 只保留核心业务接口。区分标准：响应数据是用于渲染选项列表的，就是 UI 接口，自动化不需要。

**2. 参数传递**

找清接口之间的串联关系——哪个接口的输出是下一个的输入。串联方式有三种：
- 直接提取（`$.data.id` → 变量）
- 计算获得（取列表第一条、匹配唯一字段）
- 环境变量（手动填的值）

**3. 异常处理**

自动化最容易出问题的地方不是正常流程，是异常场景：接口报错时变量没设置、轮询超时、Cookie 过期。每条链路至少覆盖：
- 接口返回非 200 时的兜底逻辑
- 轮询的退出条件（成功 + 超时）
- 变量类型兼容（字符串/数组互转）
