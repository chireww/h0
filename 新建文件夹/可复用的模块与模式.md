## 一、Python 层：两个脚本共用的函数

OSS 上传和飞书云文档上传都用 Python 写了一遍，核心模块可以抽成公共库。

### 1. `api_post` — 统一请求封装（两个脚本完全一致）

```python
def api_post(path: str, body: object, timeout: int = 60) -> dict:
    """封装 POST，遇连接错误自动重试"""
    for retry in range(3):
        try:
            resp = requests.post(
                f"{BASE_URL}{path}",
                headers=HEADERS,
                json=body,
                verify=False,
                timeout=timeout,
            )
            resp.raise_for_status()
            try:
                return resp.json()
            except json.JSONDecodeError:
                return {"raw_status": resp.status_code, "text": resp.text}
        except (requests.ConnectionError, requests.Timeout,
                requests.HTTPError) as e:
            if retry < 2:
                time.sleep((retry + 1) * 3)
                continue
            raise
```

用到的地方：**所有 POST 请求**。两个脚本各自拷了一份一模一样的，下次写新脚本直接复用。

### 2. `log` — 日志函数

```python
def log(msg: str, level: str = "INFO"):
    ts = datetime.now().strftime("%H:%M:%S")
    print(f"[{ts}] [{level}] {msg}")
```

### 3. HEADERS 配置模式

```python
HEADERS = {
    "Cookie": f"KB_SESSION_ID={KB_SESSION_ID}",
    "Authorization": AUTHORIZATION,
    "Content-Type": "application/json",
    "User-Agent": "OSS-BatchUpload/1.0",
}
requests.packages.urllib3.disable_warnings()
```

认证信息都从 `input()` 获取，Header 用同一套模板。

### 4. 并发上传模式

```python
with ThreadPoolExecutor(max_workers=MAX_WORKERS) as pool:
    futures = []
    for i, item in enumerate(items):
        fut = pool.submit(worker_function, item, i)
        futures.append(fut)
        time.sleep(0.5)  # 错峰
    for future in as_completed(futures):
        results.append(future.result())
```

下次有批量操作的场景，改 `worker_function` 和 `items` 就行。

### 5. 轮询模式

```python
def poll_until_done(check_func, timeout=120, interval=3):
    deadline = time.time() + timeout
    while time.time() < deadline:
        result = check_func()
        if result["done"]:
            return result
        time.sleep(interval)
    raise TimeoutError("轮询超时")
```

OSS 和飞书两个脚本的轮询逻辑完全一样（查 list → 比状态 → 等或退出），只是 `check_func` 不同。

## 二、Apifox 层：三个场景通用的脚本片段

### 1. 请求头配置（所有接口统一）

```
Authorization: {{authorization}}
Cookie: KB_SESSION_ID={{KB_SESSION_ID}}
Content-Type: application/json
```

配合「全局 Cookie 参数」启用，自动管理 `acw_tc` 等 WAF Cookie。

### 2. 环境变量（三个自动化共用基础 4 个）

| 变量 | 用到的地方 |
|------|-----------|
| `authorization` | 全部 |
| `KB_SESSION_ID` | 全部 |
| `project_id` | 全部 |
| `feishu_url` / `Wikifeishu_url` | 飞书上传 |
| `kb_name` | 审批 |
| `task_id` | 审批（自动提取） |

### 3. 后置操作模板

**断言**（每个接口都能用）：
```
$.msg 等于 success
```

**提取器**（一步提取，不用写脚本）：
```
$.data[*].token → tokens
$.data.fileIds  → file_ids
```

**自定义脚本保护头**（任何用 `response.data` 的地方都要加）：
```javascript
const res = pm.response.json();
if (res.code !== "00000" || !res.data) {
    pm.test("接口响应异常", () => {
        pm.expect(res.code).to.equal("00000");
        pm.expect(res.data).not.to.be.null;
    });
    return;
}
// 安全使用 res.data
```

### 4. 类型安全取值

Apifox 的变量存取不保证类型，数组可能变成字符串。任何从变量取数组的地方都加这个：
```javascript
let arr = pm.variables.get("xxx");
if (typeof arr === "string") {
    try { arr = JSON.parse(arr); } catch (e) { arr = []; }
}
arr = Array.isArray(arr) ? arr : [];
```

用到的地方：审批流程取 `file_ids`、飞书流程取 `coverArray`、汇总脚本取所有数组变量。

### 5. ForEach 累积模式（防覆盖）

```javascript
// ForEach 内不要直接 set（会被下次迭代覆盖）
// 用累积变量：
let current = JSON.parse(pm.variables.get("next_tokens") || "[]");
let accumulated = JSON.parse(pm.variables.get("accumulated_next_tokens") || "[]");
accumulated = accumulated.concat(current);
pm.variables.set("accumulated_next_tokens", JSON.stringify(accumulated));

// ForEach 结束后统一取出：
var result = pm.variables.get("accumulated_next_tokens") || "[]";
pm.variables.set("next_tokens", result);
```

用到的地方：飞书 BFS 遍历、任何需要 ForEach 内累加数据的场景。

### 6. 数据驱动的配置模式

```
环境变量定义变量名（留空）
     ↓
CSV 文件填入实际值
     ↓
Apifox 数据驱动自动替换
```

用到的地方：Wiki 上传（`Wikifeishu_url` 批量替换）。

## 三、跨工具的方法论（最值钱的部分）

### 1. 接口筛选法

```
F12 抓到的 10 个接口 → 逐个看响应
  ├─ 响应是下拉框选项 → ❌ UI 渲染
  ├─ 响应是搜索结果 → ❌ UI 渲染
  ├─ 响应是表格分页 → ❌ UI 渲染
  └─ 响应是业务数据/状态变更 → ✅ 保留
```

所有 4 个自动化都用这套判定标准，平均筛掉 60% 的接口。

### 2. 参数串联分析

```
① 画一条线把所有接口串起来
② 在每个接口箭头上标注传了什么参数给下一个
③ 找不到串联字段的接口 → 两个可能：
    - 中间漏了接口（回到 F12 补抓）
    - 参数来自环境变量/前端状态（直接写死或手动填）
```

### 3. 踩坑检查清单

每次新写一个自动化，先过一遍：

- [ ] Cookie Header 有没有手写死？→ 用全局 Cookie 管理
- [ ] `project_id`/`dir_id` 是字符串还是数字？→ 看 F12 真实请求体
- [ ] 有没有 ForEach？→ 变量会不会被覆盖
- [ ] 有没有轮询？→ 次数和间隔够不够
- [ ] 后置脚本 `response.data` 有没有 null 保护？
- [ ] 环境变量配完有没有测试能不能取到值？
